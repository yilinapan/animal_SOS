# Animal SOS — 現階段決策與待辦（2026-06-11）

## 已定案的決策

1. **產品定位**：Threads 帳號為產品本體，網站為後勤支援（延伸閱讀、信任背書、資源目錄、Meta 審核材料）。
2. **初期回覆模式**：AI 爬文/分類/擬稿 → 人工決定是否回覆 → 人工貼出。零 API、零 Meta 審核、零封禁風險。自動化問題延後到有實際數據再決定。
3. **既有研究 dossier**（docs/research/、blueprint、design、engineering）：保留作長期參考，不照其 SaaS 平台路線執行。其中模型 ID（claude-fable-5）為占位符，實作時需替換為真實 Claude API 模型 ID。
4. **不主動聯絡救援單位**：知識庫只引用公開發布的聯絡資訊，並照抄原文條件。
5. **保育類紅線**：一律「不要碰、聯絡主管機關」，不提供任何處置教學（野生動物保育法）。

## 優先順序

### P0 — 需求驗證（7 天人工觀察）【進行中】

寫程式之前先確認需求存在。每天固定 1–2 個時段、各 10–15 分鐘，在 Threads 搜尋下列關鍵字，記錄到表格：

**關鍵字組**：撿到鳥、幼鳥、雛鳥、路殺、蛇 怎麼辦、受傷 貓、受傷 狗、動物 救援、撿到 動物、鳥 撞、死掉 動物

**每篇記錄欄位**：
- 貼文連結、發文時間、發現時間
- 案型（幼鳥／路殺／蛇／受傷犬貓／大體處理／走失／其他）
- 地區（可判斷的話）
- 是否附照片
- 發文後 2 小時內是否獲得「可行動的正確資訊」（缺口判定）
- 既有回覆品質（沒人理／錯誤建議／部分正確／完整正確）
- 原 po 後續（自行解決／持續求助／無下文）

**判定指標（7 天後 go / no-go）**：
- 量：平均每天 ≥5 篇真實求助文 → 及格；<3 篇 → 重新評估
- **缺口率（最重要）**：≥40% 的求助文在 2 小時內沒有獲得可行動的正確資訊 → 有明確市場缺口
- 案型集中度：前 5 案型（幼鳥、路殺、蛇、受傷犬貓、大體）涵蓋 ≥70% → 知識庫範圍正確

**加值測試（選做）**：用個人帳號手動回覆 5–10 篇（按知識庫口吻），記錄原 po 回應率。這直接驗證核心互動是否成立，比純數量統計更有說服力。

**副產品**：收集到的 50–100 篇真實貼文 = 未來 AI 分類器的測試集。

### P2 — Firecrawl 必爬清單（驗證通過後執行，依序）

1. **台灣救援資源現況**：22 縣市動保處公告聯絡頁、路殺社、特生中心急救站、野灣、台北鳥會（WBST）、ARTT、TSPCA、各地鳥會 → 抓現行有效的電話、LINE、開放時段、收件條件 → 知識庫 v0 初始資料
2. **大體處理**：六都環保局清潔隊死亡動物收運頁面（既有研究完全遺漏的案型）
3. **Threads / Meta 開發者文件**：API quota、權限分級（`threads_keyword_search` 進階權限）、開發模式 vs 正式審核差異 → 校準階段 2 之後的時程

備註：本遠端環境目前看不到 Firecrawl connector（可能只裝在本機 Claude）；不阻塞，必要時可用內建 WebFetch 替代。

### P3 — 回覆工作台（P0 驗證通過後才動工）

私有後台：貼上貼文文字/照片 → AI 分類 + 擬稿（模板 + 資料庫組裝，禁止自由生成聯絡資訊）→ 人工確認 → 複製貼回 Threads。
